# Addendum к ТЗ «Синтетический член жюри Silver Mercury»

Документ фиксирует принятые изменения к исходному ТЗ. Применяется поверх существующего текста ТЗ и имеет приоритет в случае противоречий.

Дата: 2026-04-24.

---

## 1. Изменения в scope

### 1.1. Исключение L1 (лонглист-гейта)
На вход платформы подаются только кейсы, **уже прошедшие первый раунд отбора** (шортлист-кандидаты) — ~100 кейсов.
- Модуль L1 и весь связанный код удаляются.
- Платформа выполняет только L2: расчёт оценок по критериям блока, итоговый балл, медальную категорию и аргументацию.
- Правило `results_required` больше не ведёт к отказу в оценке; вместо этого оно капирует критерий Results (≤3 при отсутствии доказательств).

### 1.2. Итоговый формат оценки
- Балл 1–10 по каждому критерию блока (из методики XXVII).
- Взвешенный итоговый балл 1–10.
- Медальная категория согласно регламенту XXVII: longlist 0–2.99 / shortlist 3–4.99 / bronze 5–6.99 / silver 7–8.99 / gold ≥9.
- Аргументация: strengths, weaknesses, обоснование по каждому критерию, цитаты из evidence с привязкой к источнику (страница PDF / таймкод видео / поле формы).

## 2. Стек генерации аватара

### 2.1. Отмена Kling AI 2.5
Kling — генеративная text-to-video модель, непригодная для повторяющегося персонажа: между клипами возникает drift лица, голоса, освещения и кадрирования. Для 100 вердиктов аудитория увидит «разного ведущего».

### 2.2. Новый стек
- **Рендеринг видео вердикта:** HeyGen Video API (не-интерактивный).
  - Один аватар, созданный один раз (stock или custom), используется на все 100 кейсов.
  - Голос: один клонированный голос через HeyGen (или ElevenLabs voice ID, интегрированный в HeyGen).
  - Script → MP4, латентность ~1–3 минуты на клип, асинхронно.
  - Рендер 100 клипов выполняется **заранее** (ночь перед сессией), не в live-режиме.
- **Участие в Zoom:** Recall.ai bot.
  - Бот присоединяется к Zoom-комнате как отдельный участник («AI Juror»).
  - Видео/аудио MP4 вердикта передаётся в Zoom через Recall.ai Output Media API.
  - Никакого screen-share с ноутбука модератора.

### 2.3. Обоснование изменения
- Повторяемость персонажа: face/voice/фон идентичны во всех клипах.
- Lip-sync надёжный: HeyGen синхронизирует рот под фактический аудиотрек, а не генерирует его.
- Воспроизводимость: один и тот же script всегда даст один и тот же клип (важно для аудита).
- Нет live-галлюцинаций и нет on-camera латентности — модератор одобряет script до рендера.
- Путь апгрейда: если в будущем потребуется live-Q&A, HeyGen Streaming Avatar совместим с тем же avatar_id без смены стека.

## 3. Архитектура с несколькими параллельными сессиями

### 3.1. Требование
Одновременно может проходить **3–5 Zoom-сессий**, по одной на команду модераторов. Аватар одинаковый (appearance + voice) во всех.

### 3.2. Архитектура
```
Scoring engine (общий)
       │
       ▼
Генератор script (общий)
       │
       ▼
HITL-одобрение script модератором
       │
       ▼
HeyGen Video API → MP4 cache (общий, keyed by case_id)
       │
       ▼
┌──────── Moderator Console (multi-tenant, RBAC) ────────┐
│  Moderator 1 → Session A → Bot Orchestrator → Recall Bot A → Zoom Meeting A
│  Moderator 2 → Session B → Bot Orchestrator → Recall Bot B → Zoom Meeting B
│  ...
│  Moderator 5 → Session E → Bot Orchestrator → Recall Bot E → Zoom Meeting E
└────────────────────────────────────────────────────────────┘
```

### 3.3. Ключевые принципы
- **Одна библиотека рендеров** — MP4 переиспользуются любой сессией, рендер не зависит от числа комнат.
- **Один бот на сессию** — независимые состояния, никакого cross-talk.
- **Кейсы 1:1 приписаны к сессиям** — исключает коллизию одновременного проигрывания одного и того же вердикта в двух комнатах.
- **`session_id` + `moderator_id` во всех логах и API** — заложить в схему с первого дня.

### 3.4. Масштаб и стоимость
Рост с 1 до 5 параллельных сессий:
- HeyGen рендер: **без изменений** (клипы кэшированы).
- Recall.ai: линейно (~$3–6/час на бота × 5 = ~$15–30/сессия 6 часов).
- Moderator Console: требует multi-tenant UI + RBAC (дополнительные ~1–2 недели разработки).
- Bot Orchestrator: новый сервис, ~1 неделя.

## 4. Контроль модератора

### 4.1. Moderator Console (отдельное web-приложение, не внутри Zoom)
- Очередь одобренных кейсов текущей сессии с кнопкой `Play`.
- Управление проигрыванием: `Play` / `Pause` / `Stop` / `Repeat` / `Skip to next`.
- HITL-шлюз: script вердикта показывается модератору до рендера; `Approve → render → queue` либо `Edit → re-generate`.
- Отображение статусов: `rendering`, `ready`, `playing`, `done`, `failed`.
- Журнал действий модератора с timestamps (для аудита).

### 4.2. RBAC
- Роли: `admin` (управление сессиями и назначениями), `moderator` (управление своей сессией), `observer` (просмотр без действий).
- SSO или email+пароль с 2FA (требование фестиваля).

### 4.3. Аудит-журнал
Каждая запись: `session_id`, `moderator_id`, `case_id`, `action`, `timestamp`, `input_hash`, `config_hash`, `prompt_hash`, `model_id`, `mp4_hash`.

## 5. Что остаётся без изменений

- Пайплайн ingestion (PDF + OCR + vision-LLM + Whisper для видео).
- Evidence Index с привязкой цитат к источникам.
- Генерация обоснования по критериям блока с обязательной ссылкой на `evidence_id`.
- Методика XXVII как основа: 12 блоков, веса критериев, медальные пороги 3/5/7/9.
- Формула социальных критериев для социально-ориентированных номинаций.
- Anonymity scanner (flag, не автодисквалификация).

## 6. Что удалить из исходного ТЗ

- Упоминания Kling AI / Kling 2.5 в контексте рендеринга аватара.
- Формулировки про screen-share видео с ноутбука модератора как способ показа аватара.
- L1 как самостоятельный этап; все упоминания «longlist gate» / «лонг / не лонг» в платформе.
